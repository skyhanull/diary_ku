# 콘텐츠 스타일

이 문서는 UI 문구 톤과 상태 메시지 규칙을 정리한다.

## 기본 톤
- 따뜻하고 조용하다
- 친절하지만 과장하지 않는다
- 기능 설명보다 사용자를 안심시키는 방향을 우선한다
- 지나치게 제품 마케팅 문구처럼 쓰지 않는다

## 문장 스타일
- 짧고 자연스러운 한국어를 사용한다
- 기술적 내부 용어를 사용자 문구에 그대로 노출하지 않는다
- 에러도 공격적이거나 차가운 어조를 피한다

## 버튼 라벨
- 명령형이되 부드럽게 쓴다
- `저장`, `편지 공유`, `다시 시도`, `닫기`처럼 짧은 동사를 우선한다
- 지나치게 장황한 버튼 문구는 피한다

## 저장 상태 문구
현재 에디터는 아래 계열 문구를 사용한다.
- `불러오는 중...`
- `저장 중...`
- `자동 저장 중...`
- `저장에 실패했어요.`
- `저장 안 된 변경사항이 있어요.`
- `모든 변경사항이 저장됐어요.`

### 규칙
- 저장 흐름은 차분하고 설명적인 문구를 유지한다
- 성공했다고 과하게 축하하지 않는다
- 실패 메시지는 다시 시도 가능성을 남긴다

## 오류 문구
- 원인을 확신할 수 없으면 단정하지 않는다
- 사용자가 할 수 있는 다음 행동을 암시한다
- 예: `문제가 발생했어요. 다시 시도해주세요.`

## 빈 상태 문구
- 허전함보다 의도된 정적을 만든다
- 짧은 설명 + 다음 행동 하나가 기본 패턴이다
- 감성은 유지하되 행동이 흐려지지 않게 쓴다

## 입력 placeholder
- 지나치게 설명문처럼 길지 않게 유지한다
- 예시를 주더라도 실제 입력을 방해하지 않는다

## 기능별 톤 차이

### 홈
- 안내형
- 오늘의 기록과 일정 정리에 어울리는 톤

### 에디터
- 조작 보조형
- 저장, 선택, 편집 상태를 분명하게 전달

### 채팅
- 대화형
- 너무 가볍게 장난치기보다 편안한 동행 톤

### 공유 편지
- 가장 감성적인 영역
- 다만 감상문처럼 길어지기보다 읽기 쉬워야 한다

## Do
- 짧고 부드럽게 쓴다
- 상태 변화가 사용자에게 무엇을 의미하는지 알려준다
- 같은 행동에는 같은 용어를 반복 사용한다

## Don't
- 한 기능 안에서 `저장`, `업로드`, `반영`을 섞어 쓰지 않는다
- 에러 문구를 지나치게 개발자 중심으로 쓰지 않는다
- 감성 때문에 중요한 행동 문구를 모호하게 만들지 않는다
