# Future Enhancements Plan

> 포트폴리오 완성도를 높이기 위해 **나중에 추가할 기능**을 정리한 문서다.
> 지금 코드베이스에 자연스럽게 얹히는(=억지 기능이 아닌) 것 위주로 골랐다.
> 우선순위: **1) 의미 기반 검색** → **2) 실시간 협업 편집**

## 1) 의미 기반(시맨틱) 일기 검색

### 목적
현재 아카이브 검색은 **키워드 매칭**(`src/features/archive/lib/archive-search.ts`)이다.
이를 **뜻으로 찾는 검색**으로 확장한다. 예: "비 오는 날 우울했던 기록"으로 검색하면
정확한 단어가 없어도 의미가 가까운 일기를 찾아준다.

### 왜 이 프로젝트에 맞는가
- RAG 챗을 만들면서 **이미 임베딩 인프라가 다 깔려 있다.**
  - 임베딩 생성: Voyage AI `voyage-3-lite` (`src/app/api/diary/embed/route.ts`)
  - 저장: `diary_entries.embedding` (pgvector `vector(512)`)
  - 유사도 검색 RPC: `match_diary_entries(query_embedding, match_threshold, match_count)` (이미 존재)
- 즉 **검색 UI에서 이 RPC를 재사용**하면 되므로 신규 인프라가 거의 필요 없다.

### 구현 계획 (순서)
1. **쿼리 임베딩 엔드포인트 정리**
   - 검색어를 Voyage로 임베딩하는 경로 확보. `embed` 라우트를 재사용하거나
     검색 전용 `POST /api/diary/search` 라우트를 추가(입력: 검색어 → 출력: 매칭된 일기 목록).
   - 서버에서 `match_diary_entries` RPC 호출 (chat 라우트의 호출부와 동일 패턴).
2. **하이브리드 랭킹**
   - 키워드 점수(`archive-search.ts`)와 시맨틱 유사도 점수를 **가중 합산**해 정렬.
   - 완전 키워드 대체가 아니라 "키워드 + 의미" 결합이 실무적으로 자연스럽다.
3. **검색 UI 연결**
   - 아카이브 검색 입력에 "의미 검색" 토글 또는 자동 하이브리드 적용.
   - 결과에 유사도/매칭 사유를 살짝 노출하면 설득력↑.
4. **엣지 케이스**
   - 임베딩이 아직 없는 일기(과거 데이터) 처리 → 키워드 결과로 폴백.
   - 검색어가 너무 짧으면 시맨틱 스킵.

### DB 변경
- 원칙적으로 **없음**(컬럼·RPC 이미 존재).
- 성능이 필요하면 `embedding`에 `ivfflat` / `hnsw` 인덱스 추가 검토.

### 리스크 / 주의
- Voyage API 호출 비용·지연 → 검색은 디바운스 + 캐시.
- 과거 일기 임베딩 백필(backfill) 스크립트 한 번 필요할 수 있음.

### 면접 어필 포인트
- "벡터 임베딩 / pgvector / 코사인 유사도 / 하이브리드 랭킹"을 **실제로** 다뤘다고 말할 수 있음.
- 스토리: "키워드 검색의 한계 → 이미 있는 RAG 임베딩 재활용 → 키워드+시맨틱 하이브리드".

### 예상 규모
- 며칠 수준 (인프라 재활용).

## 2) 실시간 협업 편집

### 목적
편지지(캔버스)를 **여러 명이 동시에** 보고 편집한다.
최소한 다른 사람의 **커서/선택 상태 표시(presence)**, 나아가 아이템 이동/추가 실시간 반영.

### 왜 이 프로젝트에 맞는가
- 지금까지 가장 큰 약점으로 지적된 **"혼자 만든 CRUD라 협업·동시성 경험을 못 보여준다"**를 정면으로 메운다.
- 이미 관련 경험이 코드에 있다:
  - 캔버스 좌표계(`EDITOR_PAGE_WIDTH/HEIGHT`) — 위치 동기화의 기준이 잡혀 있음.
  - 자동저장 동시성 처리(`useEditorPersistenceActions`의 `saveFlowControllerRef`, latest-only) — race 처리 경험.

### 단계적 구현 계획 (작게 시작 → 확장)
**Phase A — Presence (가장 먼저, 임팩트 대비 쉬움)**
- Supabase Realtime `presence`로 접속자 목록 + 각자의 커서 좌표 브로드캐스트.
- 캔버스에 다른 사용자 커서/이름 오버레이 렌더.
- 편집 충돌 없음(그냥 표시만) → 안전하게 첫 성과.

**Phase B — 아이템 실시간 반영**
- 아이템 이동/추가/삭제 이벤트를 Realtime broadcast.
- 수신 측은 **optimistic update + last-write-wins(LWW)**로 반영.
- 서버 저장은 기존 diff 방식(`saveEditorSession`) 유지, 충돌은 updated_at 기준 정리.

**Phase C — (선택, 고급) 충돌 해결 고도화**
- 동시에 같은 아이템을 옮길 때의 튐 현상 → 아이템 단위 버전/락 또는 CRDT 도입 검토.
- 여기까지 가면 난이도 급상승. 포트폴리오용으론 Phase A~B로도 충분.

### DB / 인프라 변경
- Supabase Realtime 채널(엔트리별 room) 사용.
- 아이템 테이블에 `updated_at` 기반 충돌 판정(이미 있는 컬럼 활용).
- 협업 세션 참여자 표시는 클라이언트 상태로 충분(영속 불필요).

### 리스크 / 주의
- 제대로 하면 CRDT/OT까지 가서 어렵다 → **Phase A(presence)부터** 반드시 작게 시작.
- 네트워크 이벤트 폭주 → 커서/이동 브로드캐스트는 throttle 필수.
- 인증·권한: 편지 원본은 작성자 소유 → 협업 허용 범위(초대 링크 등) 정책 먼저 정의.

### 면접 어필 포인트
- "실시간 동기화 / presence / optimistic UI / 충돌 해결(LWW)" — 주니어가 잘 못 보여주는 영역.
- 기존 자동저장 동시성 처리와 연결해 **일관된 "동시성" 서사** 완성.

### 예상 규모
- Phase A: 수일 / Phase B: 1~2주 / Phase C: 별도 도전 과제.

## 3) 에이전트형 AI 친구 (tool-use)

### 목적
현재 AI 친구는 **RAG(유사도 top-k 검색)** 로만 동작한다. 질문과 비슷한 과거 일기 몇 개를
컨텍스트로 넣어 답할 뿐이라, "6월에 운동한 날 며칠이야?", "가장 자주 쓴 감정은?" 같은
**집계·분석형 질문**에는 약하다.
이를 **도구 호출(tool-use / function calling)** 로 확장해, LLM이 필요할 때 DB를 직접 조회·집계하게 한다.

### 왜 이 프로젝트에 맞는가
- 기존 `/api/diary/chat` RAG 파이프라인을 **업그레이드**하는 구조라 새 인프라가 거의 없다.
- "RAG top-k"는 이제 기본기 취급 → **tool-use/에이전트 루프**는 LLM 채용에서 가장 많이 묻는 스킬.
- RAG의 한계를 인지하고 넘어섰다는 **성숙도**를 보여줄 수 있다.
- 방금 만든 2.5개월치 시드 데이터(다양한 감정·태그) 덕분에 분석형 질문이 **데모에서 실제로 답이 나온다**.

### 도구(tool) 후보
| 도구 | 설명 | 반환 |
|------|------|------|
| `search_entries` | 의미/키워드로 일기 검색 (기존 RAG 재사용) | 관련 일기 목록 |
| `count_entries` | 기간·감정·태그 조건으로 개수 집계 | 숫자 |
| `get_entry_by_date` | 특정 날짜 일기 조회 | 일기 1건 |
| `list_entries` | 기간/필터로 목록 조회 (감정·태그별) | 일기 목록 |
| `mood_stats` | 기간별 감정 분포 집계 | 감정별 카운트 |

### 구현 계획 (순서)
1. **도구 스키마 정의** — 위 도구들을 function-calling 스키마(JSON schema)로 정의.
2. **도구 실행 계층** — 각 도구를 Supabase 쿼리로 매핑. **반드시 `user_id`로 스코프**(본인 데이터만).
3. **에이전트 루프** — LLM 응답에 tool_call이 있으면 실행 → 결과를 다시 모델에 전달 → 최종 답변.
   - 무한 루프 방지: 최대 tool 호출 횟수 상한(예: 5회).
4. **스트리밍 유지** — 도구 턴이 끝난 뒤 최종 답변만 스트리밍하거나, 중간 상태를 UI에 표시.
5. **폴백** — 도구가 필요 없는 일반 대화는 기존 RAG 흐름 그대로.

### 모델 / 인프라
- Groq Llama 3.3-70B의 tool-use 지원 여부 착수 시 재확인(현재 지원). 미지원 시 tool-use 지원 모델로 교체.
- DB 접근은 기존 `getAuthenticatedSupabase` 재사용 → RLS + `user_id` 이중 방어.

### 리스크 / 주의
- **스트리밍 + tool-calling 동시 처리**가 단순 RAG보다 복잡함(멀티턴 루프).
- 도구가 DB를 조회하므로 **데이터 격리(본인 것만)** 가 최우선. SQL 인젝션 없이 파라미터 바인딩.
- 도구 호출 남발로 인한 지연·비용 → 호출 상한 + 필요할 때만 호출하도록 프롬프트 유도.

### 면접 어필 포인트
- "function calling / 에이전트 루프 / 도구 스키마 설계 / RAG 한계 극복" — LLM 직무 핵심 질문 영역.
- 스토리: "RAG는 집계형 질문에 약하다 → 도구 호출로 DB를 직접 질의 → 정확한 분석 답변".

### 예상 규모
- 중~높음 (1~2주). 도구 3~5개부터 시작해 점진 확장.

> 참고: 3번을 먼저 만들어두면 1번(시맨틱 검색)은 `search_entries` 도구로, 2번 이후의 주간 회고 등도
> 이 도구들을 재사용할 수 있어 확장이 쉬워진다.
